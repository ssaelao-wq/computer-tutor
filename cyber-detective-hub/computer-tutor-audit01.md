## The Headline Finding

The parent `computer-tutor/` folder contains a **complete, high-quality, per-level curriculum document set** — `Computer_Tutor_AIEra_Curriculum_Overview.md` plus one detailed `L{1-4}-Computer_Tutor_AIEra_Curriculum.md` per level (each with full session plans, board-lesson scripts, Socratic debugging scenarios, real-world ethics cases, homework, and tutor answer keys), and a matching `L1-Computer_Tutor_Teacher_Guideline.md`. This is materially more developed than what's actually implemented in the app.

**Only Level 1 has been synced into the app**, and even then only partially:
- `src/projectTasksData.js` (the Project Journal data) matches `L1-Computer_Tutor_AIEra_Curriculum.md` almost line-for-line — session 4 is "JS Variable Registry" ↔ doc's "Tracking Game State: JS Variables & Math"; session 5 is "Keyboard Control Interfaces" ↔ "Steering the Car: JS Keydown Event Listeners"; session 9 is "Timer Loops & Animations" ↔ "The Racing Game Loop: Timers & Animations"; and so on through session 12. **This part of the app was clearly built correctly from the source doc.**
- `src/curriculumData.js` (the Curriculum Guide data — what students actually see as session content), by contrast, was **not** synced from that same doc for sessions 4-12. It instead contains generic, non-game topics — a thermostat, a pet feeder, Blockly block-matching, a stopwatch, a habit tracker, a checkout interface — that appear nowhere in `L1-Computer_Tutor_AIEra_Curriculum.md`. Side-by-side for Session 4 alone:

  | Field | `curriculumData.js` (`l1-s4`) | `L1-Computer_Tutor_AIEra_Curriculum.md` (Session 4) |
  |---|---|---|
  | Title | "Decisions, Decisions" | "Tracking Game State: JS Variables & Math" |
  | Core activity | "Thermostat safety override: Program logical conditions to regulate sector climate." | Declare `carX`, `speed`, `score`, `gameActive` and perform math updates on them. |
  | Matches Project Journal (`projectTasksData.js` `l1-s4`, "JS Variable Registry")? | No | Yes |

  This pattern repeats for sessions 4, 5, 6, 7, 8, 9, 10, 11, and 12 — nine of Level 1's twelve sessions.

- **Levels 2-4 haven't been synced into the app at all.** `L2-Computer_Tutor_AIEra_Curriculum.md` is a fully-written **"Mars Colony Defense Game"** curriculum (Canvas rendering, sprite arrays, laser batteries, alien swarms, keyboard matrices, colony shields, async leaderboard APIs — 13 sessions). `L3-Computer_Tutor_AIEra_Curriculum.md` is a fully-written **"Cyberpunk Hacker Arena"** curriculum (database schemas, "hacking sign-ups," player inventories, a "Hacking Console," RLS, deployment — 14 sessions). Neither theme — sprites/lasers/aliens for L2, hacking console/player inventory for L3 — appears anywhere in `src/curriculumData.js` or `src/projectTasksData.js`. The app's L2/L3 content is instead generic programming-literacy material unconnected to either theme.

**Reframing:** this is not primarily a "write more content" problem. The content already exists, is well-written, and (for L1) has already been proven out once by whoever built `projectTasksData.js`. The gap is an **integration/sync gap**: someone needs to port `L1-Computer_Tutor_AIEra_Curriculum.md` sessions 4-12 into `curriculumData.js`, and port `L2`/`L3`/`L4-Computer_Tutor_AIEra_Curriculum.md` into both `curriculumData.js` and `projectTasksData.js`, the same way L1's Project Journal was already done correctly. That materially changes the effort estimate and priority order versus my first pass at this audit (see Recommendations).

## A. Curriculum Content Quality — Source Docs vs. App

Every one of the 51 session entries in `curriculumData.js` is structurally complete (`objectives`, `warmUp`, `miniLesson`, `coreActivity`, `handsOn`, `homework`, `ethics`, `adaptations` all populated), and the writing itself is competent — good hooks, plausible classroom activities, ethics questions tied to the technical topic. Taken in isolation, no session in `curriculumData.js` reads as broken or low-effort.

The problem only appears when you compare it against the `L{1-4}-Computer_Tutor_AIEra_Curriculum.md` docs, which are a *different, later, more detailed* curriculum draft — richer per session (a full minute-by-minute lesson timeline, a "Build Phase (AI Assisted)" step, a "Socratic Debugging" seeded-bug scenario, a real-world ethics case citation, and a full tutor answer key), and, critically, actually on-theme for the level. `curriculumData.js` was evidently written against an earlier, generic draft and simply never updated when the per-level docs were finalized. The `Computer_Tutor_AIEra_Curriculum_Overview.md` even documents a specific pedagogical structure — "The 3-Part Session Flow" (Concept Learning → Seeded-Bug Exercises → Game Target Integration) and a Bronze/Silver/Gold tiering system for the game-integration step — that isn't reflected in `curriculumData.js`'s fields at all.

## B. Level Theme Alignment

Per `Computer_Tutor_AIEra_Curriculum_Overview.md` §4 (Program Structure table), the four level themes are: **L1 "Racing Car Game," L2 "Mars Colony Defense," L3 "Cyberpunk Hacker Arena," L4 "The Software Engineer."**

- **Level 1**: `L1-Computer_Tutor_AIEra_Curriculum.md` is fully racing-themed end-to-end (sessions 1-12). `projectTasksData.js` matches it. `curriculumData.js` matches it only for sessions 1-3; sessions 4-12 drift into generic/domestic examples (see Headline Finding).
- **Level 2**: `L2-Computer_Tutor_AIEra_Curriculum.md` session titles are entirely Mars-Colony/space-shooter themed: "Initializing the Defense Arena: Canvas & Coordinates," "The Laser Battery: Arrays of Sprites," "Alien Swarms: Nested Arrays & Movement Deltas," "Colony Shields: Destructible Grid Matrices," "Submitting High Scores to the Cloud Leaderboard," etc. **None of this appears in `curriculumData.js` or `projectTasksData.js`.** The app's L2 content (restaurant/API analogies, Lego version-control, vending-machine debugging) is generic code-literacy material with no connection to the Mars theme.
- **Level 3**: `L3-Computer_Tutor_AIEra_Curriculum.md` is Cyberpunk-Hacker-Arena themed: "User Authentication & Hacking Sign-Ups," "Relational Queries: Retrieving Player Inventories," "Deploying the Hacking Console to the Cloud," "The Hacker Arena Integration Sprint." The app's L3 content in `curriculumData.js` covers the same underlying CS topics (databases, APIs, RLS, security) but with generic framing (a library schema, a restaurant analogy, a ride-sharing app) rather than the hacker-arena/player-inventory skin the source doc uses.
- **Level 4**: Both `curriculumData.js` and `L4-Computer_Tutor_AIEra_Curriculum.md` independently land on generic, non-game "professional software engineer" framing (testing, CI/CD, monitoring, launch) — they roughly agree with each other. The mismatch here is against `PRJ_KNOWLEDGE.md`, which still describes L4 as a "Mission Control Dashboard" real-time monitoring theme (see Finding #6 / Documentation Set Consistency below).

`CAMPAIGN_THEMES` in `App.jsx` (a teacher-selectable global reskin: cyberpunk metropolis / Mars colonization / magic academy, applied identically across all 4 levels) is a **third, independent theming concept** that matches none of the above — e.g. its "mars" variant reskins *Level 1* (which should always be Racing Car Game) into a spacesuit/oxygen-regulator narrative, while its own Level 2 entry is titled "Operation: AI Copilot," not Mars Colony Defense. This needs to be resolved as a product decision — keep `CAMPAIGN_THEMES` and drop the fixed-per-level theme idea, or retire `CAMPAIGN_THEMES` in favor of the fixed themes the curriculum docs already commit to — before investing in the sync work described above, since building against the wrong mechanism means redoing it.

## D. Racing Car Project Tasks (`projectTasksData.js` / Project Journal)

This remains the strongest asset in the current build, and the audit now has direct evidence of *why*: it was built faithfully from `L1-Computer_Tutor_AIEra_Curriculum.md`. All 12 sessions have complete task records — `objectives`, a `planSpecs` block, a `promptGuide`, a `codeReviewGuide` with genuine Socratic questions, a `testCasesGuide`, and an `iterationGuide` — and the progression (HTML skeleton → CSS layout → variables → keyboard input → boundary guards → obstacle loops → modular functions → animation loop → collision detection → HUD/restart → polish) matches the source doc's 11-Part Roadmap exactly. This is the template to replicate for L2-L4, not something that needs rework itself.

## E. UI/UX & Workflow (Code-Level Read)

Navigation is a fixed left sidebar with 6-7 client-state tabs, no router. Notable points:
- "Quest Files" (`cases` tab) shows only sessions present in `CAMPAIGN_THEMES` (12 total, explicitly labelled "Playable Exercise Quests Only" at `App.jsx:2086`), while "Curriculum Guide" (`curriculum` tab) shows all 51 `CURRICULUM_DATA` sessions with an "Exercise" button on every one — implying a parity between the two views that doesn't exist.
- Four largely-independent session-selector states (`selectedSessionId`, `curriculumSessionId`, `sandboxSessionId`, and the Project Journal's own selector) are bridged only by explicit "Open Exercise"/"Exercise" buttons that call `loadTemplate()` before switching tabs — fine for the common browse-then-launch path, but there's no reverse sync if a student navigates via the sidebar directly.
- Sidebar/header label mismatches ("Quest Files" vs. "Quest Board," "Curriculum Guide" vs. "Curriculum Syllabus Catalog"), and the internal tab id `sandbox` is user-facing-labelled "Exercises Journal" while a *separate* tab id `journal` is "Project Journal" — easy to confuse in code, tickets, or conversation.
- Role gating (Admin Panel only for `role === 'teacher'`) and level-based auto-routing on login both look correctly implemented.

## F. Documentation Set Consistency

Beyond the app itself, the source documents don't fully agree with each other, which matters because `PRJ_KNOWLEDGE.md` is the one currently designated (in this project's `CLAUDE.md`) as the source of truth to consult before product decisions:
- `PRJ_KNOWLEDGE.md` describes L4 as "Mission Control Dashboard" (real-time collaborative monitoring); `Computer_Tutor_AIEra_Curriculum_Overview.md` and `L4-Computer_Tutor_AIEra_Curriculum.md` both call it "The Software Engineer" with no game skin. The latter two are more detailed and appear to be the newer, more fully-developed design — worth a pass to reconcile `PRJ_KNOWLEDGE.md`'s language, or confirm with whoever owns curriculum design which is current.
- `Computer_Tutor_Teacher_Guideline.md` (the "general" filename) and `L1-Computer_Tutor_Teacher_Guideline.md` are byte-identical — the "general" guideline is actually just Level 1 content under a different name. There's no teacher guideline yet for L2-L4.
- `curriculumData.js`'s module groupings for Level 1 (e.g. "Module 2: Visual Structures & Logical Loops," "Module 4: Thinking Like a Tester") don't match either `L1-Computer_Tutor_AIEra_Curriculum.md`'s module boundaries (Sessions 1-3 / 4-6 / 7-9 / 10-11 / 12) or the Overview doc's summary — another marker that `curriculumData.js` was forked from an earlier draft and not reconciled since.

## G. Live Browser Walkthrough — Project Journal (2026-07-08)

Driven live as `student-a` (`npm run dev`, Playwright headless Chromium, no console errors observed at any step). Focused on the Project Journal tab per request.

**Workflow structure is genuinely good where content exists.** Each session in the Project Journal opens to a tabbed 5-step strip that maps directly onto the framework: **1. Plan & Design → 2. Write AI Prompt → 3. Review & Explain → 4. Test & Break → 5. Iterate & Improve.** For L1 sessions 1-4 (the synced ones), every tab is pre-seeded with real, milestone-specific guidance — e.g. Step 3 for Session 3 asks *"What happens if `#game-track` loses `position: relative`? Which coordinate box will the browser calculate offsets against?"* This is a faithful implementation of the Socratic-audit pedagogy described in the curriculum docs, not a generic form with the session name swapped in.

**Verified: guide text is real `placeholder`, not a submittable pre-filled value.** Inspecting the DOM directly (not just visually) confirmed `textarea.value === ""` with the guide copy living in the `placeholder` attribute. A student cannot get credit by leaving the guide questions untouched — they have to actually type something, and the guide text disappears as they do. This is a small thing but it's the kind of detail that determines whether "audit the AI's output" is actually enforced or just suggested.

**Commenting on AI-generated code — answering the specific question asked.** Step 3 ("Review & Explain") already has exactly this shape: a left textarea to paste the AI's code, a right textarea (seeded with the Socratic questions above) for the student's own audit notes. What it does *not* have is line-by-line/inline annotation — it's one comment box against one pasted blob, so a student could write a single vague paragraph about a 40-line file and technically complete the step. Checking `TODO.md` confirms this is a known, already-scoped gap: v1 is deliberately whole-file copy-paste ("forces the student to read every line"), and line-by-line annotation is explicitly planned for v2, gated behind Git-diff integration once projects outgrow a single HTML file (matching the level-gated review-mechanism design in `PRJ_KNOWLEDGE.md` §7). So: real gap, but not a surprise — it's already on the roadmap, and this audit's contribution is confirming it's worth prioritizing (see section H).

**Two things found only by clicking, not by reading code:**
- **The title/content mismatch is visible today, not hypothetical.** Clicking "Initialize Project Journal" on Session 4 produces a screen headed "L1 S4: Decisions, Decisions" directly above a milestone panel reading "RACING CAR PROJECT — PART 3 · JS Variable Registry" with correctly car-themed content underneath. A real student sees this contradiction on their first visit to an unsynced session — it's not a subtle inconsistency buried in data files, it's a visibly broken page.
- **The prompt-version badge ("v1") doesn't do anything.** It's clearly meant to be a version indicator/selector (the `journal_versions` table backing it supports multiple entries per journal, and Step 5 is explicitly about comparing iterations), but clicking it produces no dropdown, history list, or diff. As it stands, a student finishing Step 5 ("what did you learn from this prompting cycle") has no way to actually look back at their own previous prompt to compare against.

## H. Big Picture: Does the Platform Serve the AI-Era Learning Objective?

Stepping back from individual bugs: `Computer_Tutor_AIEra_Curriculum_Overview.md` states the mission plainly — shift students from "Syntax Writers" to "System Architects," using AI as the "automated food processor" while the student supplies specification, judgment, and audit. The mechanism for this is the **Prompt Journal Protocol**, and the doc is explicit that it should be **"the Primary Grade Metric"** — grading shifts from "does the code run" (which AI now guarantees) to "how good was the student's specification, and how rigorous was their audit of what came back." Everything else in the curriculum — the 5-Step loop, the "Fallible Intern" mindset, the "Break It" mentality — is designed to feed that one artifact.

Judged against that specific bet, here's where the platform stands:

- **The workflow skeleton is right.** The 5-tab structure, the seeded Socratic questions, the real-placeholder enforcement, and the milestone framing tied to a tangible, always-visible project (the racing car, not an abstract "exercise 7") are all faithful to the design. This part doesn't need reinvention — it needs the coverage fix (sections A-D) to reach L1 sessions 5-12 and Levels 2-4.
- **The platform currently has no way to actually grade the artifact it just told itself is the primary grade metric.** The Admin Panel can move a student's XP up or down, but a teacher has no screen anywhere to read what a student actually wrote in Plan & Design, Review & Explain, or Iterate & Improve. If the Prompt Journal is supposed to be graded on the quality of the specification and the depth of the audit notes (per the Overview doc's own "AI-Era Pedagogical Best Practices" §G.4), the current app structurally cannot support that grading model — a teacher would have to go around the platform (open the database directly, or stand behind the student) to actually assess the thing the curriculum says should be assessed. This is the single highest-leverage gap relative to the platform's own stated goal, arguably ahead of the content-sync work in sections A-D, because even perfectly-synced L2-L4 content would inherit the same blind spot.
- **The automated Sandbox `validate()` checks are keyword-matching, not specification-quality checks** (e.g. a passing AI-prompt exercise just needs the strings `'create'`, `'div'`, `'id'`, `'game-track'` to appear somewhere in the textarea — `App.jsx:557-560`). That's a reasonable, cheap bootstrap for absolute-beginner HTML/CSS syntax exercises (L1 S1-S3), where the thing being taught really is "did you use the right keywords in the right shape." It stops being a good proxy the moment the skill being assessed *is* specification quality itself (L2's "Prompting as a Specification Document" module, all of L3-L4) — a student could satisfy most of these checks with a technically-keyword-complete but lazy, ambiguous prompt, which is exactly the failure mode the curriculum is trying to train students out of. `PRJ_KNOWLEDGE.md`'s v2 "TA Auditor" concept (an embedded LLM that cross-checks a student's prompt/journal against their own design plan and the session's objectives) is the right long-term fix for both this and the teacher-visibility gap above — it's worth pulling forward in priority precisely because it's the piece that make the platform's core grading claim actually true at classroom scale, rather than a nice-to-have.
- **The "Ethical Lens" has curriculum content but no durable, gradable home in the student's own workflow.** Every session in `curriculumData.js` has a real, well-written `ethics` field, and it surfaces in the Curriculum Guide's read-only session-plan view — but the Project Journal's 5 steps (Plan/Prompt/Review/Test/Iterate) don't include an ethics or reflection field of their own. As written, ethics is something a teacher narrates and a student reads, not something a student writes into their own portfolio artifact the way their Socratic audit notes are. Given the curriculum explicitly calls ethics "integrated into every level, not bolted on," it's worth asking whether it should leave a trace in the Journal too — even a lightweight one.
- **Where the design is already right, it's worth affirming, not just critiquing.** The Hybrid Delivery Model's split — external AI IDEs (Cursor, ChatGPT) do code generation; the platform tracks methodology and never executes or generates code itself — is a sound, deliberately scoped architectural choice that keeps the platform from competing with tools it can't out-build, and it's followed consistently everywhere I looked (no code-execution sandbox, no in-app AI calls). Similarly, the level-gated review escalation (copy-paste → git diff → PR → CI/CD) correctly treats "how you review code" as its own skill ladder rather than a one-size-fits-all feature. The job ahead is mostly finishing what's already been designed well, not redesigning it.

**Net read:** the platform's biggest risk isn't that any single feature is missing — it's that without completion gating (Finding #5), teacher-facing journal visibility, or quality-aware grading, the full 5-step loop can currently be satisfied by typing a few sentences into five boxes and clicking through, which would quietly defeat the "cognitive resistance" the entire curriculum is built around. None of that is a criticism of what's been built so far — L1's implemented sessions show the team knows how to build the rigorous version. It's a sequencing observation: **content coverage (sections A-D) makes the platform look complete; grading/audit rigor (this section) is what would make it actually work as designed.** Worth weighing both when prioritizing.

## J. Systematic Cross-Reference: Document ↔ Software

**#J-1 — `db_setup.sql` and `vite_db.sql` cannot actually run this application if followed as instructed.** `README.md` and `ARCHITECTURE.md` both point to these files for local database setup, and `PRJ_KNOWLEDGE.md` §8 tells L3 students to run `vite_db.sql`/`db_setup.sql` to create their local database. Diffing their `user_profile` table against what `server.cjs` actually requires at runtime:

| Column | `db_setup.sql` / `vite_db.sql` | `server.cjs` (`createTables()`) |
|---|---|---|
| `username` | ❌ missing | ✅ required (`UNIQUE`, used for login) |
| `password` | ❌ missing | ✅ required (used for login) |
| `role` | ❌ missing | ✅ required (drives student/teacher gating) |
| `student_level` | ❌ missing | ✅ required (drives level routing) |

A database created fresh from `db_setup.sql` would have a `user_profile` table with no `username`/`password` columns at all — `POST /api/auth/login` would fail immediately (`server.cjs:233` queries `WHERE username = $1`). `server.cjs` papers over this for *itself* via idempotent `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` migrations baked into `createTables()`, but a student or teacher who runs only the standalone `.sql` file — exactly as instructed by `PRJ_KNOWLEDGE.md` §8's Level 3 setup checklist — gets a database the app cannot log into. This is a concrete, reproducible setup-breaking bug, not just a documentation staleness note.

**#J-2 — `AGENTS.md` references a grading file that doesn't exist.** Rule #7 says "NEVER change Thai 4.0 grading logic without checking `lib/grades.js` first" — there is no `lib/` directory and no `grades.js` anywhere in this repo (confirmed via `Glob`). Either the rule is aspirational (grading logic hasn't been built yet) or it refers to a file from a different project/an earlier restructuring. As written it's a dangling reference that would confuse an agent instructed to "check `lib/grades.js` first" when no such file exists.

**#J-3 — Confirms and sharpens the earlier MySQL/Postgres finding from `CLAUDE.md`.** `server.cjs` uses `pg.Pool` exclusively (`require('pg')`), `.env` configures a Supabase `DATABASE_URL`, and `id SERIAL` / `$1` placeholder syntax throughout `createTables()` — all Postgres-only syntax that would not run against MySQL at all. `ARCHITECTURE.md` §2.2 and §3.1 both still say "MySQL2/pg" and "Type: Relational SQL Database (MySQL)." This isn't just stale wording — `db_setup.sql`/`vite_db.sql` are genuinely MySQL-dialect files (`AUTO_INCREMENT`, backtick-free but MySQL-flavored `INT`/`VARCHAR` defaults) that would need translation to run against the Postgres database this app actually uses, on top of the missing-columns problem in #J-1.

**#J-4 — `package.json`'s claimed stack matches the software; the docs that describe it as MySQL-based don't.** `package.json` lists `pg` and `mysql2` as *both* being dependencies (`mysql2: ^3.22.5`, `pg: ^8.22.0`) — but `server.cjs` only ever imports `pg`. `mysql2` is an unused dependency sitting in `package.json`, presumably left over from when the app targeted MySQL directly, before the Supabase/Postgres migration. Worth removing if confirmed dead, or documenting why it's still needed.

## K. Systematic Cross-Reference: Software ↔ Software

**#K-1 — The header/content mismatch (Finding #10) is the flagship example, but the general pattern is: `curriculumData.js` and `projectTasksData.js` are keyed by the same `l{level}-s{n}` IDs and are supposed to describe the same session, and for L1 sessions 4-12 they don't.** Already covered in the Headline Finding and section A; noted here for completeness of the cross-reference sweep.

**#K-2 — Two different, disagreeing rank-tier systems coexist in `App.jsx`.** The main rank badge (sidebar/dashboard) uses `getRank()` (`App.jsx:1214-1219`), a 4-tier system: *Novice Decoder* (0-399), *Junior Investigator* (400-799), *Senior Auditor* (800-1199), *Master Architect* (1200+). The Leaderboard tab, a few thousand lines away, computes its own rank label inline (`App.jsx:4090`): `scorePoints >= 1200 ? 'Master' : scorePoints >= 800 ? 'Senior' : 'Junior'` — a **3**-tier system with different names and no distinction between 0-399 and 400-799. A student with 200 XP is "Novice Decoder" on their own dashboard and "Junior" on the leaderboard. This is a straightforward duplicated-logic bug — `getRank()` should be the single source of truth and the Leaderboard should call it instead of re-deriving its own thresholds.

**#K-3 — `INITIAL_JOURNAL` (`App.jsx:113-156`) is dead code** (confirmed by the `oxlint` warning noted in Finding #9's neighborhood) containing a single hardcoded seed journal entry ("L1 S1: Household IPO Blueprint," dated `2026-06-24`) that duplicates content also hardcoded into a `server.cjs` migration (`UPDATE journal_versions SET code = 'Household IPO Blueprint...'`, `server.cjs:~199`). Two different files carry the same one-off seed text for the same purpose, neither is wired to the other, and the frontend copy isn't used at all. Low-severity, but worth deleting when someone's next in this area — it's a second, independent trace of the same "microwave" example flagged as off-theme in Findings #1/#5.

**#K-4 — The `CAMPAIGN_THEMES` reskin data (`App.jsx:158-527`) only overrides 5 of Level 1's 12 sessions, 2 of Level 2's 13, 2 of Level 3's 14, and 3 of Level 4's 12** — for every session it doesn't cover, the UI silently falls back to `curriculumData.js`'s un-reskinned title/homework.
